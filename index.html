Arquitectura de Software

La arquitectura de software es el diseño estructural de un sistema, que define cómo interactúan sus componentes entre sí para cumplir con los requisitos funcionales y no funcionales.

Propósito de la arquitectura

  • Organizar el código y los componentes. Facilitar la escalabilidad, mantenimiento y evolución del sistema. Garantizar que el software cumpla con los objetivos del negocio.
  • Facilita mantenimiento y evolución
  • Responde a objetivos del negocio

Componentes principales

  • Módulos: Autenticación, usuarios, etc.
  • Interfaces: API, sockets
  • Patrones: MVC, capas, microkernel

Principios clave

  • Modularidad : Divide el sistema en partes independientes.
  • Abstracción : Oculta detalles complejos detrás de interfaces simples.
  • Escalabilidad : Permite que el sistema crezca sin perder rendimiento.
  • Reusabilidad : Componentes que pueden usarse en diferentes contextos.

Ejemplo en desarrollo web

En una aplicación web como un comercio electrónico, la arquitectura podría incluir: Una capa de presentación (frontend con HTML/CSS/JavaScript). Una capa de lógica de negocio (backend con Node.js o Django). Una capa de datos (base de datos como MySQL o MongoDB).

Arquitectura Monolítica

Una arquitectura monolítica es un enfoque donde todos los componentes de una aplicación están integrados en un solo sistema o ejecutable . Todo (frontend, backend, base de datos, lógica de negocio) se desarrolla, despliega y ejecuta como una unidad.

Monolito: Caracteristicas

  • ✔ Código unificado : Toda la lógica está en un solo proyecto.
  • ✔ Despliegue único : Un solo archivo o servidor contiene toda la aplicación.
  • ✔ Base de datos compartida : Todos los módulos acceden a la misma base de datos.
  • ✔ Simplicidad : Fácil de desarrollar y probar en etapas iniciales.
  • ✔ Despliegue sencillo : Un solo servidor o contenedor es suficiente.
  • ✔ Depuración directa : Todo está en un solo lugar, lo que facilita encontrar errores.


  • ✘ Escalabilidad limitada : Difícil de escalar horizontalmente (agregar más servidores).
  • ✘ Mantenimiento complejo : A medida que crece, el código se vuelve difícil de gestionar.
  • ✘ Dependencia fuerte : Un cambio en un módulo puede romper toda la aplicación.

    • Arquitectura de Microservicios

    La arquitectura de microservicios divide una aplicación en servicios pequeños, independientes y especializados , cada uno encargado de una función específica. Estos servicios se comunican a través de APIs (generalmente REST o gRPC) y pueden desarrollarse, desplegarse y escalarse por separado.

    Microservicios: Pros y Contras

    • ✔ Escalable, flexible, resiliente, Innovador


    • ✘ Complejo, la latencia entre servicios, costo operativo, requiere herramientas modernas

    Arquitectura Cliente-Servidor

    La arquitectura cliente-servidor es un modelo donde un cliente (como un navegador web o una aplicación móvil) solicita recursos o servicios a un servidor , que procesa las solicitudes y devuelve respuestas. Es la base de la mayoría de las aplicaciones web modernas.

    Arquitectura cliente servidor: Pros y Contras

    • ✔ Centralización, Escalabilidad, Seguridad


    • ✘ Dependencia del servidor, Latencia, Carga en el sevidor,

    Resumen: ¿cuándo usar cada una?

    • Monolito: Proyectos pequeños
    • Microservicios: Escalabilidad y equipos grandes
    • Cliente-servidor: Modelo universal para apps web
